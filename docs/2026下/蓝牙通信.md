## 1. 前置知识：BLE 核心概念

### 1.1. BLE 是什么？

**BLE（Bluetooth Low Energy，蓝牙低功耗）** 是蓝牙 4.0 标准引入的低功耗无线通信协议，与经典蓝牙（BR/EDR）并列，专为短距离、低频率、小数据量的场景设计。

| 对比项 | 经典蓝牙 | BLE |
|--------|---------|-----|
| 功耗 | 高 | 极低（纽扣电池可用数年） |
| 传输速率 | 高（音频/文件） | 低（传感器/信标） |
| 连接延迟 | 秒级 | 毫秒级 |
| 典型场景 | 耳机、音箱 | 心率带、门锁、碰一碰 |

微信小游戏通过 `wx.openBluetoothAdapter` 开启 BLE 适配器，本项目的碰一碰功能完全基于 BLE 实现。

---

### 1.2. 中心设备 vs 外围设备

BLE 通信中，两端角色是**不对称**的：

```mermaid
graph LR
    subgraph "外围设备 Peripheral（广播方）"
        P1["📡 持续广播 Advertising Packet<br/>包含设备名、UUID、payload"]
    end
    subgraph "中心设备 Central（扫描方）"
        C1["🔍 扫描 Scan<br/>监听附近广播包"]
    end
    P1 -->|"BLE 广播（单向，无需配对）"| C1
    C1 -->|"发起 GATT 连接（双向）"| P1
```

| 角色 | 英文 | 职责 | 微信 API |
|------|------|------|---------|
| **外围设备** | Peripheral | 主动广播自身信息，等待连接 | `wx.createBLEPeripheralServer()` |
| **中心设备** | Central | 扫描周围广播，主动发起连接 | `wx.startBluetoothDevicesDiscovery()` |

#### 1.2.1. 本项目的特殊之处：双角色同时运行

每台手机**同时扮演两个角色**：

```mermaid
graph TD
    subgraph "手机 A（你的手机）"
        A_P["Peripheral 角色<br/>广播自己的 tempId+petId"]
        A_C["Central 角色<br/>扫描周围设备"]
    end
    subgraph "手机 B（对方手机）"
        B_P["Peripheral 角色<br/>广播自己的 tempId+petId"]
        B_C["Central 角色<br/>扫描周围设备"]
    end
    A_P -->|"广播"| B_C
    B_P -->|"广播"| A_C
```

**核心代码**（`wx-adapter.ts`）：
```ts
// iOS 需要分别以两种 mode 各调一次
wx.openBluetoothAdapter({ mode: 'central' })   // 开启扫描能力
wx.openBluetoothAdapter({ mode: 'peripheral' }) // 开启广播能力
```

> **⚠️ iOS 限制**：Android 调一次 `openBluetoothAdapter` 即可同时具备两种能力；iOS 必须分别调用两次，且必须在用户点击事件回调中触发，否则报错。

### 1.3. GATT 是什么？

**GATT（Generic Attribute Profile，通用属性协议）** 是 BLE 在建立连接后，用于**读写数据**的标准协议框架。

#### 1.3.1. GATT 的层级结构

```mermaid
graph TD
    Device["BLE 设备"] --> Server["GATT Server（服务端，通常是 Peripheral）"]
    Server --> S1["Service 服务<br/>UUID: 0000FEED-...<br/>（逻辑功能分组）"]
    Server --> S2["Service 服务<br/>UUID: 0000180D-...<br/>（心率服务，标准服务举例）"]
    S1 --> C1["Characteristic 特征值<br/>UUID: 0000BEEF-...<br/>（实际存放数据的地方）"]
    C1 --> V["Value 值<br/>存放 payload 字节数组"]
    C1 --> D["Descriptor 描述符<br/>描述特征值的属性（可通知/可读/可写）"]
```

| 概念 | 类比 | 本项目的值 |
|------|------|-----------|
| **Service（服务）** | 一个功能模块 | `0000FEED-0000-1000-8000-00805F9B34FB` |
| **Characteristic（特征值）** | 模块里的一个数据字段 | `0000BEEF-0000-1000-8000-00805F9B34FB` |
| **Value（值）** | 字段里存的数据 | `NLDY4WFJW6\|3f4nv5bgfe9ad`（tempId\|petId） |

#### 1.3.2. GATT 的三种数据交互方式

```mermaid
sequenceDiagram
    participant C as Central（读取方）
    participant P as Peripheral（数据方）

    Note over C,P: 方式1：Read（主动读）
    C->>P: wx.readBLECharacteristicValue()
    P-->>C: 返回 Value

    Note over C,P: 方式2：Write（主动写）
    C->>P: wx.writeBLECharacteristicValue(0x01)
    Note over P: onCharacteristicWriteRequest 触发

    Note over C,P: 方式3：Notify（订阅推送）⭐ 本项目核心
    C->>P: wx.notifyBLECharacteristicValueChange(state:true)
    Note over C: 订阅成功，等待推送
    P->>C: server.writeCharacteristicValue(needNotify:true)
    Note over C: onBLECharacteristicValueChange 回调触发
```

> **本项目为什么用 Write 触发 Notify，而不是直接 Read？**
> 因为微信 iOS SDK 的 `onCharacteristicReadRequest` 回调**不会触发**（iOS CoreBluetooth 的 Peripheral 模式在微信 SDK 中未完整暴露）。所以改用：Central 先订阅 Notify，再发一个 Write `0x01` 作为"敲门砖"，Peripheral 收到 Write 请求后主动 Notify 推送 payload。

### 1.4. deviceId 是什么？

`deviceId` 是**微信 BLE API 为每台扫描到的蓝牙设备分配的唯一标识符**，由微信底层生成，开发者无法控制其格式。

#### 1.4.1. 不同平台的格式

| 平台 | deviceId 格式 | 示例 |
|------|-------------|------|
| **iOS** | UUID 格式（大写） | `A1B2C3D4-E5F6-7890-ABCD-EF1234567890` |
| **Android** | MAC 地址格式（大写，冒号分隔） | `AA:BB:CC:DD:EE:FF` |

> **⚠️ 重要**：iOS 出于隐私保护，**不暴露真实 MAC 地址**，每次重启蓝牙后 UUID 可能变化。Android 直接使用 MAC 地址，相对稳定。

#### 1.4.2. deviceId 在本项目中的作用

```mermaid
flowchart TD
    A["wx.onBluetoothDeviceFound 回调<br/>携带 deviceId"] --> B["存入 deviceIdToPeer Map<br/>deviceId → tempId"]
    B --> C{"需要 GATT 补齐？"}
    C -->|"是（iOS 扫 Android）"| D["wx.createBLEConnection(deviceId)<br/>用 deviceId 建立 GATT 连接"]
    C -->|"否"| E["直接用广播里的 tempId 上报"]
    D --> F["wx.getBLEDeviceServices(deviceId)"]
    F --> G["wx.getBLEDeviceCharacteristics(deviceId, serviceId)"]
    G --> H["wx.notifyBLECharacteristicValueChange(deviceId, ...)"]
    H --> I["wx.writeBLECharacteristicValue(deviceId, ...)"]
    I --> J["收到 payload，补齐 tempId+petId"]
    J --> K["wx.closeBLEConnection(deviceId)"]
```

**核心代码**（`BumpService.ts`）：
```ts
// deviceId 贯穿整个 GATT 连接流程
private async _fetchPeerPetIdViaGatt(peerTempId: string, deviceId: string): Promise<string> {
    // 1. 建立连接
    await wx.createBLEConnection({ deviceId });
    // 2. 协商 MTU（避免 payload 被 20 字节截断）
    await wx.setBLEMTU({ deviceId, mtu: 512 });
    // 3. 发现服务
    const { services } = await wx.getBLEDeviceServices({ deviceId });
    // 4. 发现特征值
    const { characteristics } = await wx.getBLEDeviceCharacteristics({ deviceId, serviceId });
    // 5. 订阅 Notify
    await wx.notifyBLECharacteristicValueChange({ deviceId, serviceId, characteristicId, state: true });
    // 6. 发送 Write 触发对端推送
    await wx.writeBLECharacteristicValue({ deviceId, serviceId, characteristicId, value: triggerBuf });
    // 7. 等待 onBLECharacteristicValueChange 回调
    // 8. 关闭连接
    await wx.closeBLEConnection({ deviceId });
}
```

#### 1.4.3. deviceId 的去重作用


### 4.2. 广播阶段（Peripheral 角色）

每台设备通过 `wx.createBLEPeripheralServer` 创建外围设备服务器，向外广播身份信息。

#### 4.2.1. 广播包结构

```
BLE Advertising Packet（31字节上限）
├── AD Type 0x01  Flags（3字节）
├── AD Type 0x07  Complete List of 128-bit UUIDs
│     └── 0000FEED-0000-1000-8000-00805F9B34FB（18字节）
├── AD Type 0x09  Complete Local Name
│     └── "NLDY4WFJW6_3f4nv5bgfe9ad"（25字节）
└── AD Type 0xFF  Manufacturer Specific Data
      └── mfrId(0xFEED, 2B) + payload字符串
```

**核心代码**（`wx-adapter.ts`）：

```ts
// 只广播 [FEED] 一个 UUID，释放广播包空间给完整 deviceName
server.startAdvertising({
  advertiseRequest: {
    deviceName: bumpName,                    // "NLDY4WFJW6_3f4nv5bgfe9ad"
    serviceUuids: [params.serviceUuid],      // ["0000FEED-..."]
    manufacturerData: [{ manufacturerId: BUMP_MANUFACTURER_ID, manufacturerSpecificData }],
    serviceData: [{ uuid: params.serviceUuid, data: valueBuf }],
  }
})
```

> **为什么去掉 `BUMP_` 前缀？**
> 早期同时广播 3 个 UUID（FEED + BE01 + BE02），广播包空间耗尽，iOS 系统将 `localName` 截断至仅剩 3 字节（`BUMP_NLDY4WFJW6_3f4nv5bgfe9ad` → `NLD`）。去掉前缀并只广播 `[FEED]` 一个 UUID 后，`localName` 有足够空间放下完整 25 字节 payload。

#### 4.2.2. 广播 Fallback 机制

```mermaid
flowchart LR
    A["首选<br/>deviceName + FEED + manufacturerData + serviceData"] -->|失败| B
    B["备选1<br/>deviceName + FEED + manufacturerData"] -->|失败| C
    C["备选2<br/>deviceName + FEED"] -->|失败| D
    D["备选3<br/>只广播 FEED UUID"]
```

#### 4.2.3. GATT Peripheral 三种响应机制

注册 GATT 服务后，iOS Peripheral 通过三种方式响应对端的数据请求：

```mermaid
flowchart TD
    A["对端 Central 连接"] --> B{"触发方式"}
    B -->|"onBLEPeripheralConnectionStateChanged<br/>（iOS SDK 有时不触发）"| C["延迟 1.5s<br/>server.writeCharacteristicValue<br/>needNotify:true"]
    B -->|"onCharacteristicWriteRequest<br/>（核心机制）"| D["立即 notify 推送 payload"]
    B -->|"onCharacteristicReadRequest<br/>（部分机型支持）"| E["writeCharacteristicValue<br/>needNotify:false 响应 read"]
```

**核心代码**（`wx-adapter.ts`）：

```ts
// 核心：监听 write 请求，收到后立即 notify 推送 payload
// 绕过 onBLEPeripheralConnectionStateChanged 在微信 iOS SDK 里不触发的问题
if (typeof wx.onCharacteristicWriteRequest === 'function') {
  wx.onCharacteristicWriteRequest((res: any) => {
    server.writeCharacteristicValue({
      serviceId: params.serviceUuid,
      characteristicId: params.characteristicUuid,
      value: valueBuf,
      needNotify: true,   // 通过 Notification 推送给订阅方
    });
  });
}
```

### 4.3. 扫描阶段（Central 角色）

每台设备通过 `wx.startBluetoothDevicesDiscovery` 扫描附近 BLE 设备。

#### 4.3.1. 扫描参数

```ts
wx.startBluetoothDevicesDiscovery({
  allowDuplicatesKey: true,  // 允许重复上报，用于 RSSI 实时更新
  interval: 0,               // 最快扫描间隔
  // ⚠️ 不传 services 过滤！
  // 原因：iOS 广播包经微信 SDK 重新包装后 serviceUUIDs 格式可能不匹配，
  //       Android 传 services 过滤后实测完全扫不到 iOS 设备。
})
```

扫到设备后，通过 `wx.onBluetoothDeviceFound` 回调触发 `_onPeerFound`，再用 `matchesService()` 做软过滤：

```mermaid
flowchart TD
    A["wx.onBluetoothDeviceFound 回调"] --> B["matchesService() 软过滤"]
    B -->|"advertisServiceUUIDs 含 FEED"| C["✅ 是本游戏设备"]
    B -->|"serviceData key 含 FEED"| C
    B -->|"advertisData 字节序列含 FEED"| C
    B -->|"设备名格式 xxx_yyy（字母数字）"| C
    B -->|"以上均不匹配"| D["❌ 过滤掉（耳机/手表等）"]
    C --> E["parsePayload() 解析 payload"]
```

> **iOS 兜底轮询**：iOS 上 `allowDuplicatesKey: true` 有时不上报重复设备，导致 RSSI 不更新。因此每 1500ms 调一次 `wx.getBluetoothDevices()` 主动轮询。

### 4.4. Payload 解析：多通道候选池

`parsePayload()` 从多个通道并行解析，选出信息最完整的结果：

```mermaid
flowchart LR
    subgraph 通道
        A["通道A<br/>advertisServiceUUIDs<br/>BE01/BE02 tag 编码"]
        B["通道B<br/>serviceData[FEED_UUID]<br/>Android↔Android 标准"]
        C["通道C<br/>advertisData<br/>去掉2字节mfrId后解析"]
        D["通道D<br/>localName<br/>tempId_petId 格式"]
    end
    A & B & C & D --> E["候选池 candidates[]"]
    E --> F{"优先级选择"}
    F -->|"有 petId 且 tempId≥4位"| G["✅ 最优结果"]
    F -->|"仅 tempId≥4位"| H["次优结果"]
    F -->|"其他"| I["兜底结果"]
```

**核心代码**（`utils.ts`）：

```ts
// 挑最完整的：优先 (有 petId 且 tempId 长度 ≥ 4) > (仅 tempId 长度 ≥ 4) > 其他
const withPet = candidates.find((c) => !!c.petId && (c.myTempId?.length || 0) >= 4);
if (withPet) return withPet;
const fullTemp = candidates.find((c) => (c.myTempId?.length || 0) >= 4);
if (fullTemp) return fullTemp;
return candidates[0];
```

### 4.5. GATT 主动连接补齐（方案 C）

当广播里解析不出完整 payload 时（尤其是 **iOS 扫 Android** 场景），触发 GATT 主动连接读取。

```mermaid
sequenceDiagram
    participant A as iOS Central
    participant B as Android Peripheral

    Note over A: parsePayload 失败，生成占位 tempId=~ZYBPHKKM
    A->>B: wx.createBLEConnection(deviceId)
    Note over A: 失败时延迟1s重试（errCode=10003/133 常见）
    A->>B: wx.setBLEMTU({ mtu: 512 })
    Note over A,B: 协商 MTU，避免 payload 被默认 20 字节截断
    A->>B: wx.getBLEDeviceServices()
    A->>B: wx.getBLEDeviceCharacteristics()
    A->>B: wx.notifyBLECharacteristicValueChange({ state: true })
    Note over A: 订阅 Notification
    A->>B: wx.writeBLECharacteristicValue(0x01)
    Note over B: onCharacteristicWriteRequest 触发
    B->>A: server.writeCharacteristicValue(needNotify:true)
    Note over A: wx.onBLECharacteristicValueChange 回调
    A->>A: 解析 payload="ZYBPHKKMO9J|3gme3wdjj4ab9"
    A->>B: wx.closeBLEConnection()
    Note over A: 迁移占位 tempId → 真实 tempId<br/>dispatch BumpPeerLost(~ZYBPHKKM)<br/>dispatch BumpPeerFound(ZYBPHKKMO9J)
```

**核心代码**（`BumpService.ts`）：

```ts
// 广播里没拿到 petId 时，发起主动 GATT 连接读取对端特征值
if (rawPetId) {
  void this._fetchPeerNickAndUpdate(peerTempId, rawPetId);
} else if (deviceId && this.bt) {
  void this._fetchPeerPetIdViaGatt(peerTempId, deviceId);
}
```

**保护机制**：
- **并发去重**：同一 `deviceId` 的并发调用返回同一个 Promise
- **失败熔断**：累计失败 ≥ 3 次后直接返回空串，不再重试
- **超时兜底**：整个流程 8 秒超时，超时后强制 `wx.closeBLEConnection`

### 4.6. 雷达状态机

```mermaid
stateDiagram-v2
    [*] --> Idle
    Idle --> Starting : start()
    Starting --> Scanning : BumpStart API 成功
    Starting --> Failed : BumpStart API 失败

    Scanning --> Scanning : 扫到设备 / 设备离开
    Scanning --> Reporting : bumpPeer(peerTempId)
    Reporting --> WaitingMatch : BumpReport 返回 waiting
    Reporting --> Matched : BumpReport 返回 matched
    Reporting --> Scanning : BumpReport 失败（软失败）

    Matched --> Scanning : 奖励弹窗关闭，继续碰下一只
    Scanning --> Idle : stop()
    Failed --> [*]
```

## 6. 关键踩坑与解决方案

### 6.1. iOS 广播 `localName` 被截断

**现象**：`NLDY4WFJW6_3f4nv5bgfe9ad`（25字节）被截断为 `NLD`（3字节）

**根因**：BLE 广播包总大小 31 字节，同时广播 3 个 128-bit UUID（FEED + BE01 + BE02）占用 54 字节，超出限制后 iOS 系统自动截断 `localName`。

**解决**：只广播 `[FEED]` 一个 UUID，释放 36 字节空间给 `localName`。

---

### 6.2. Android 传 `services` 过滤后扫不到 iOS 设备

**现象**：`wx.startBluetoothDevicesDiscovery({ services: [FEED_UUID] })` 后，Android 完全扫不到 iOS 设备。

**根因**：iOS 广播包经微信 SDK 重新包装后，`advertisServiceUUIDs` 里的 UUID 格式（大小写、短格式）与过滤条件不完全匹配，系统层直接过滤掉。

**解决**：双端都不传 `services` 过滤，改用 `matchesService()` 软过滤。

---

### 6.3. iOS Peripheral 的 `onCharacteristicReadRequest` 不触发

**现象**：Android Central 调 `wx.readBLECharacteristicValue` 后，8 秒内没有任何回调，超时。

**根因**：微信 iOS SDK 未完整暴露 CoreBluetooth 的 Peripheral 模式回调，`onCharacteristicReadRequest` 在 iOS 上不会触发。

**解决**：改用 **write 触发 notify** 方案——Android Central 订阅 notify 后发送 `0x01`，iOS Peripheral 的 `onCharacteristicWriteRequest` 收到后立即 notify 推送 payload。

---

### 6.4. Android↔Android GATT 连接 errCode=10003

**现象**：第一次 `wx.createBLEConnection` 失败，errCode=10003（errno=133）。

**根因**：Android BLE 栈的已知问题，第一次连接时 GATT 层偶发初始化失败。

**解决**：失败后延迟 1 秒重试一次。

---

### 6.5. GATT 默认 MTU 导致 payload 截断

**现象**：GATT 读取到的 payload 只有 20 字节，`3gme3wdjj4ab9` 被截断为 `3gme3wdj`。

**根因**：BLE GATT 默认 MTU=23，有效载荷只有 20 字节，而完整 payload `NLDY4WFJW6|3f4nv5bgfe9ad` 约 25 字节。

**解决**：连接成功后立即调 `wx.setBLEMTU({ mtu: 512 })`，Android 实际协商值通常为 185~512 字节。

# 蓝牙碰一碰通信机制详解

## 1. 前置知识：BLE 核心概念

### 1.1. BLE 是什么？

**BLE（Bluetooth Low Energy，蓝牙低功耗）** 是蓝牙 4.0 标准引入的低功耗无线通信协议，与经典蓝牙（BR/EDR）并列，专为短距离、低频率、小数据量的场景设计。

| 对比项 | 经典蓝牙 | BLE |
|--------|---------|-----|
| 功耗 | 高 | 极低（纽扣电池可用数年） |
| 传输速率 | 高（音频/文件） | 低（传感器/信标） |
| 连接延迟 | 秒级 | 毫秒级 |
| 典型场景 | 耳机、音箱 | 心率带、门锁、碰一碰 |

微信小游戏通过 `wx.openBluetoothAdapter` 开启 BLE 适配器，本项目的碰一碰功能完全基于 BLE 实现。

---

### 1.2. 中心设备 vs 外围设备

BLE 通信中，两端角色是**不对称**的：

```mermaid
graph LR
    subgraph "外围设备 Peripheral（广播方）"
        P1["📡 持续广播 Advertising Packet<br/>包含设备名、UUID、payload"]
    end
    subgraph "中心设备 Central（扫描方）"
        C1["🔍 扫描 Scan<br/>监听附近广播包"]
    end
    P1 -->|"BLE 广播（单向，无需配对）"| C1
    C1 -->|"发起 GATT 连接（双向）"| P1
```

| 角色 | 英文 | 职责 | 微信 API |
|------|------|------|---------|
| **外围设备** | Peripheral | 主动广播自身信息，等待连接 | `wx.createBLEPeripheralServer()` |
| **中心设备** | Central | 扫描周围广播，主动发起连接 | `wx.startBluetoothDevicesDiscovery()` |

#### 1.2.1. 本项目的特殊之处：双角色同时运行

每台手机**同时扮演两个角色**：

```mermaid
graph TD
    subgraph "手机 A（你的手机）"
        A_P["Peripheral 角色<br/>广播自己的 tempId"]
        A_C["Central 角色<br/>扫描周围设备"]
    end
    subgraph "手机 B（对方手机）"
        B_P["Peripheral 角色<br/>广播自己的 tempId"]
        B_C["Central 角色<br/>扫描周围设备"]
    end
    A_P -->|"广播"| B_C
    B_P -->|"广播"| A_C
```

**核心代码**（`wx-adapter.ts`）：
```ts
// iOS 需要分别以两种 mode 各调一次
wx.openBluetoothAdapter({ mode: 'central' })   // 开启扫描能力
wx.openBluetoothAdapter({ mode: 'peripheral' }) // 开启广播能力
```

> **⚠️ iOS 限制**：Android 调一次 `openBluetoothAdapter` 即可同时具备两种能力；iOS 必须分别调用两次，且必须在用户点击事件回调中触发，否则报错。

### 1.3. GATT 是什么？

**GATT（Generic Attribute Profile，通用属性协议）** 是 BLE 在建立连接后，用于**读写数据**的标准协议框架。

#### 1.3.1. GATT 的层级结构

```mermaid
graph TD
    Device["BLE 设备"] --> Server["GATT Server（服务端，通常是 Peripheral）"]
    Server --> S1["Service 服务<br/>UUID: 0000FEED-...<br/>（逻辑功能分组）"]
    S1 --> C1["Characteristic 特征值<br/>UUID: 0000BEEF-...<br/>（实际存放数据的地方）"]
    C1 --> V["Value 值<br/>存放 payload 字节数组"]
    C1 --> D["Descriptor 描述符<br/>描述特征值的属性（可通知/可读/可写）"]
```

| 概念 | 类比 | 本项目的值 |
|------|------|-----------|
| **Service（服务）** | 一个功能模块 | `0000FEED-0000-1000-8000-00805F9B34FB` |
| **Characteristic（特征值）** | 模块里的一个数据字段 | `0000BEEF-0000-1000-8000-00805F9B34FB` |
| **Value（值）** | 字段里存的数据 | `aQ6jL3K0`（tempId，固定 8 位） |

#### 1.3.2. GATT 的数据交互方式（Write 触发 Notify）

```mermaid
sequenceDiagram
    participant C as Central（读取方）
    participant P as Peripheral（数据方）

    Note over C,P: 本项目核心：Write 触发 Notify
    C->>P: wx.notifyBLECharacteristicValueChange(state:true)
    Note over C: 订阅成功，等待推送
    C->>P: wx.writeBLECharacteristicValue(0x01)
    Note over P: onCharacteristicWriteRequest 触发
    P->>C: server.writeCharacteristicValue(needNotify:true)
    Note over C: onBLECharacteristicValueChange 回调触发
```

> **为什么用 Write 触发 Notify，而不是直接 Read？**
> 因为微信 iOS SDK 的 `onCharacteristicReadRequest` 回调**不会触发**（iOS CoreBluetooth 的 Peripheral 模式在微信 SDK 中未完整暴露）。所以改用：Central 先订阅 Notify，再发一个 Write `0x01` 作为"敲门砖"，Peripheral 收到 Write 请求后主动 Notify 推送 payload。

---

### 1.4. deviceId 是什么？

`deviceId` 是**微信 BLE API 为每台扫描到的蓝牙设备分配的唯一标识符**，由微信底层生成，开发者无法控制其格式。

| 平台 | deviceId 格式 | 示例 | 稳定性 |
|------|-------------|------|--------|
| **iOS** | UUID 格式（大写） | `A1B2C3D4-E5F6-7890-ABCD-EF1234567890` | ⚠️ 同一 session 内稳定，跨 session 可能变化 |
| **Android** | MAC 地址格式 | `AA:BB:CC:DD:EE:FF` | 相对稳定，但部分厂商有随机化 |

> **⚠️ 重要**：因为 deviceId 跨 session 不稳定（尤其 iOS），本项目的持久化缓存**不用 deviceId 做 key**，而是用后端分配的 `petId`（唯一稳定）。deviceId 仅作为会话内的临时去重标识。

### 2.1. 两种扫描模式

| 模式 | BLE 实例 | 触发时机 | 广播 | 扫描 | 缓存写入 |
|------|---------|---------|------|------|---------|
| **后台扫描** | `_bgBt` | 首页 `_afterGuide` 后常驻 | 有 tempId 才广播 | ✅ 始终 | ✅ 写入 |
| **前台扫描** | `bt` | 用户点"碰一碰"进入 | ✅ 广播 | ✅ 扫描 | ✅ 写入 |

> **关键设计**：微信小游戏 BLE 适配器全局只有一个，两个实例不能同时运行。进入前台碰一碰时先 `stopBackgroundScan()`，退出时自动恢复后台扫描。

## 4. 完整通信时序

### 4.1. 后台扫描（首页常驻）

```mermaid
sequenceDiagram
    participant Home as GooseHomeApp
    participant S as BumpService
    participant BT as BluetoothBump(_bgBt)
    participant WX as 微信 BLE API
    participant API as 后台 API

    Note over Home: _afterGuide 完成
    Home->>S: startBackgroundScan()
    S->>S: isBluetoothAuthorized()?
    alt 有 tempId（之前进过碰一碰）
        S->>BT: new BluetoothBump + start(tempId)
        BT->>WX: openAdapter + startAdvertising + startDiscovery
        Note over BT: 扫描+广播模式
    else 无 tempId（新用户/重新进入）
        S->>BT: new BluetoothBump + start(undefined)
        BT->>WX: openAdapter + startDiscovery
        Note over BT: 仅扫描模式（广播启动失败自动降级）
    end

    Note over S: 启动 tempId 定时刷新（每 4 分钟）
    loop 每 4 分钟
        S->>API: BumpStart({ act_id }) 申请新 tempId
        API-->>S: { temp_id (8位) }
        S->>BT: stop() + new BluetoothBump + start(newTempId)
        Note over BT: 用新 tempId 重启广播（保持对端能扫到有效 tempId）
    end

    loop 每 1-3s 扫到设备
        WX-->>BT: onDeviceFound
        BT-->>S: _onBgDeviceFound(dev)
        S->>S: tempId 长度校验（必须 === 8）
        S->>S: 写入 _bgPendingDevices[deviceId]
        S->>API: getPetByTempId(tempId)
        API-->>S: { pet_id, user_nick, pet.nick }
        S->>S: 迁移到 _nearbyPeerCache[petId]
        S->>S: _saveNearbyPeerCacheToStorage()
    end
```

> **tempId 有效期 5 分钟**：后台扫描每 4 分钟通过 BumpStart 重新申请 tempId 并重启广播，预留 1 分钟安全边际。首次无 tempId 时也会在第一个 4 分钟周期尝试获取（从"仅扫描"升级为"扫描+广播"模式）。

### 4.3. 设备发现与缓存（两阶段策略）

```mermaid
flowchart TD
    A["onDeviceFound 回调"] --> B["matchesService() 软过滤"]
    B -->|"不匹配 FEED UUID"| X["❌ 丢弃"]
    B -->|"匹配"| C["parsePayload() 解析 tempId"]
    C --> D{"tempId 长度 === 8？"}
    D -->|"否/为空"| E{"deviceId 在放弃集合？"}
    E -->|"是"| X
    E -->|"否"| F["GATT 补齐<br/>_fetchPeerPetIdViaGatt"]
    F -->|"成功且 tempId=8位"| G
    F -->|"失败"| H["加入 _gattGaveUpDevices<br/>不再重试"]
    D -->|"是"| G["有效 tempId"]
    G --> I{"正式缓存已有<br/>（按 deviceId 反查）？"}
    I -->|"是"| J["更新 lastSeenAt / rssi / tempId"]
    I -->|"否"| K{"pending 区已有？"}
    K -->|"是"| J
    K -->|"否"| L["新建 entry → _bgPendingDevices[deviceId]"]
```

**核心设计**：
- **会话内去重**：用 deviceId（`_bgPendingDevices` + `_findCacheEntryByDeviceId`）
- **持久化 key**：用 petId（跨 session 稳定唯一）
- **tempId 校验**：后台固定下发 8 位，非 8 位是截断/损坏的无效数据，直接过滤

## 5. 雷达状态机

```mermaid
stateDiagram-v2
    [*] --> Idle
    Idle --> Starting : start()
    Starting --> Scanning : BumpStart API 成功
    Starting --> Failed : BumpStart API 失败 / 蓝牙未开

    Scanning --> Scanning : 扫到设备 / 设备离开
    Scanning --> Reporting : bumpPeer(peerTempId)
    Reporting --> Matched : BumpReport 返回 matched / IM 推送
    Reporting --> Scanning : BumpReport 失败（软失败，err_msg toast）
    Matched --> Scanning : 奖励弹窗关闭，继续碰下一只
    Scanning --> Idle : stop()
    Failed --> Idle : 蓝牙重新开启（watchAdapterStateForRetry）
```

## 7. tempId 校验规则

后台 BumpStart 下发的 tempId **固定 8 位**（字母+数字），在以下 3 个入口做统一校验：

```ts
// 非 8 位直接过滤，不走后续流程
if (!rawTempId || rawTempId.length !== 8) {
  // 走 GATT 补齐（仅前台） 或 直接跳过（后台）
  return;
}
```

| 入口 | 校验失败行为 |
|------|------------|
| `_onPeerFound`（前台） | 走 GATT 补齐；已放弃的 deviceId 跳过 |
| `_onBgDeviceFound`（后台） | 直接 return |
| `_fetchPeerPetIdViaGatt` 回调 | 标记放弃 + return |

**效果**：
- 截断的 tempId（iOS 扫 Android，广播帧被裁剪）→ 走 GATT 补齐
- 噪音数据（非本游戏设备拼出的随机字符）→ 直接丢弃
- 避免用无效 tempId 调 `getPetByTempId`（省网络请求）

## 8. GATT 补齐与放弃机制

### 8.1. 触发条件

仅 **iOS 扫 Android** 场景：广播包里 `localName` 为空，`parsePayload` 解析不出有效 tempId。

### 8.2. 保护机制

```mermaid
flowchart TD
    A["readPeerPayload(deviceId)"] --> B{"_readFailCount >= 3?"}
    B -->|"是"| C["⚠️ 日志 + 返回空串"]
    B -->|"否"| D["GATT 连接 → setBLEMTU → subscribe notify → write 0x01 → 等 payload"]
    D -->|"成功"| E["清除 failCount"]
    D -->|"失败/超时"| F["failCount++"]
    C --> G["_gattGaveUpDevices.add(deviceId)"]
    G --> H["后续 onDeviceFound 不再对此 deviceId 调 GATT"]
```

**关键**：`onDeviceFound` 每 1-3s 周期性重复上报同一台设备。如果没有放弃机制，GATT 失败后每秒都会打日志"已达失败上限"（无限循环）。`_gattGaveUpDevices` 在入口处直接跳过，彻底止血。

### 8.3. 重置时机

`_gattGaveUpDevices` 在 `BumpService.start()` 时清空——每次新会话给设备一次重新尝试的机会。

## 9. 关键踩坑与解决方案

### 9.1. iOS 广播 `localName` 被截断

**现象**：payload 被截断为 3 字节  
**解决**：只广播 `[FEED]` 一个 UUID，释放空间给 `localName`

### 9.2. Android 传 `services` 过滤后扫不到 iOS 设备

**解决**：不传 `services` 过滤，改用 `matchesService()` 软过滤

### 9.3. iOS Peripheral 的 `onCharacteristicReadRequest` 不触发

**解决**：改用 **write 触发 notify** 方案

### 9.4. GATT 默认 MTU 导致 payload 截断

**解决**：连接后立即 `wx.setBLEMTU({ mtu: 512 })`

### 9.5. 重新进入小游戏后扫描不到别人

**根因**：`stop()` 清空 `state.myTempId`，`startBackgroundScan()` 旧版要求 tempId 非空才启动  
**解决**：允许"仅扫描不广播"模式——没有 tempId 时仍启动 discovery，只是不 advertise

### 9.6. `readPeerPayload 已达失败上限` 日志疯狂打印

**根因**：`onDeviceFound` 每 1-3s 重复触发同一 deviceId，每次都调 `readPeerPayload`  
**解决**：新增 `_gattGaveUpDevices` Set，失败后标记放弃，入口处直接跳过

### 9.7. 同一用户显示两个气泡

**根因**：deviceId 在 iOS 跨 session 可能变化，缓存用 deviceId 做 key 导致同一用户存为两条  
**解决**：缓存 key 改为 petId（唯一稳定），deviceId 仅做会话内临时去重

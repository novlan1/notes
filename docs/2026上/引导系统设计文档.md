

## 1. 概述

本引导系统采用**位掩码（Bitmask）机制**管理用户的新手引导进度，支持多步骤并行完成、独立标记、灵活查询等功能。

## 2. 核心设计原则

### 2.1. 引导与领养分离
- **真正的引导**：仅包含主页操作引导（喂食、收蛋、赚鹅粮等）
- **领养流程**：选鹅、取名、领养成功属于首次领养流程，由 `PetService.getFirstPet()` 判定
- **边界清晰**：没有宠物时弹出 AdoptPage，有宠物但引导未完成时弹出 GuidePage

### 2.2. 位掩码机制
- **幂次形式**：每个步骤使用 2 的幂次方作为位值（1, 2, 4, 8, 16...）
- **独立标记**：每个步骤可以独立完成，不受顺序限制
- **灵活查询**：通过位运算快速判断步骤完成状态

## 3. 引导步骤定义

### 3.1. 步骤常量（位掩码值）
```typescript
export const STEP_NONE = 0;                    // 0   未开始
export const STEP_FEED = 1 << 0;              // 1    喂食引导
export const STEP_COLLECT = 1 << 1;           // 2    收蛋引导
export const STEP_CLAIM_PRIZE = 1 << 2;      // 4    兑奖引导
export const STEP_EARN_FOOD = 1 << 3;        // 8    赚鹅粮引导
export const STEP_TAP = 1 << 4;              // 16   碰一碰引导

export const STEP_ALL = STEP_FEED | STEP_COLLECT | STEP_CLAIM_PRIZE | STEP_EARN_FOOD | STEP_TAP;
```

### 3.2. 展示顺序
```typescript
export const STEP_ORDER: readonly number[] = [
  STEP_FEED,
  STEP_COLLECT,
  STEP_CLAIM_PRIZE,
  STEP_EARN_FOOD,
  STEP_TAP,
];
```

**注意**：展示顺序与位值大小独立，可以按业务需要调整顺序。

## 4. 掩码操作函数

### 4.1. 判断步骤完成状态
```typescript
export function hasStep(mask: number, step: number): boolean {
  return (mask & step) === step;
}
```

### 4.2. 标记步骤完成
```typescript
export function addStep(mask: number, step: number): number {
  return mask | step;
}
```

### 4.3. 查找下一个未完成步骤
```typescript
export function nextPendingStep(mask: number): number | null {
  for (const s of STEP_ORDER) {
    if (!hasStep(mask, s)) return s;
  }
  return null;
}
```

### 4.4. 判断是否全部完成
```typescript
export function isAllFinished(mask: number): boolean {
  return (mask & STEP_ALL) === STEP_ALL;
}
```

## 5. 数据流设计

### 5.1. 引导进度获取
```typescript
// 从 PlayerService 获取当前引导进度
const guideStep = PlayerService.ins.getGuideStep();
```

### 5.2. 引导起点推断
```typescript
// 自动推断起点：从 guide_step 找出第一个未完成的步骤
const startStep = nextPendingStep(guideStep);
if (startStep === null) {
  // 全部完成，直接结束引导
  await this._finishAll();
  return;
}
```

### 5.3. 引导进度推进
```typescript
// 完成当前步骤并推进到下一个
const oldMask = PlayerService.ins.getGuideStep();
const newMask = addStep(oldMask, currentStep);

// 提交服务端
await advanceGuideStep({ step: newMask });
await PlayerService.ins.reload();

// 检查是否全部完成
if (isAllFinished(latestMask)) {
  await this._finishAll();
  return;
}

// 推进到下一个未完成步骤
const nextStep = nextPendingStep(latestMask);
this._gotoStep(nextStep);
```

## 7. 服务端兼容性

### 7.1. 位掩码模式（推荐）
- 服务端将 `guide_step` 字段当作 bitmask 解读
- 推进时使用位或操作：`guide_step |= step`
- 支持步骤的独立完成和乱序完成

### 7.2. 严格递增模式（兼容）
- 如果服务端暂时仍是严格递增模式
- 前端可以一次提交所有完成项：`advanceGuideStep(STEP_ALL)`
- 配置文件本身不依赖服务端的写策略

## 11. 扩展性设计

### 11.1. 新增引导步骤
1. 在 `GuideConfig.ts` 中添加新的步骤常量
2. 更新 `STEP_ORDER` 数组
3. 添加对应的 `STEP_META` 元数据
4. 更新 `STEP_ALL` 掩码

### 11.2. 调整引导顺序
- 直接修改 `STEP_ORDER` 数组顺序
- 不影响已完成的步骤标记

### 11.3. 条件引导
- 基于 `hasStep()` 函数判断特定步骤是否完成
- 实现条件触发的引导流程

## 12. 最佳实践

### 12.1. 避免魔法数字
- 所有业务逻辑使用导出的常量
- 禁止硬编码 1/2/4 等数字

### 12.2. 错误处理
- API 调用失败时不阻塞 UI 流转
- 使用本地掩码继续推进流程

### 12.3. 性能优化
- 复用同一 Promise 防止并发重复请求
- 使用只读快照避免不必要的状态更新

## 13. 总结

本引导系统通过位掩码机制实现了灵活、可扩展的引导进度管理，支持：
- 多步骤并行完成
- 独立的状态标记
- 灵活的查询和推进
- 良好的服务端兼容性
- 易于扩展和维护



## 14. 掩码机制的核心优势

### 14.1. **步骤独立性**
- 每个步骤可以独立完成，不受顺序限制
- 用户可以先完成"收蛋"(2)，再完成"喂食"(1)
- 服务端只需要记录完成状态，不需要关心完成顺序

### 14.2. **灵活的状态查询**
```typescript
// 快速判断特定步骤是否完成
const hasFeed = (guideStep & STEP_FEED) === STEP_FEED;
const hasCollect = (guideStep & STEP_COLLECT) === STEP_COLLECT;

// 判断是否全部完成
const allFinished = (guideStep & STEP_ALL) === STEP_ALL;
```

### 14.3. **服务端兼容性强**
- **位掩码模式**：服务端使用 `guide_step |= step` 操作
- **严格递增模式**：前端可以一次性提交 `STEP_ALL` 兼容旧版本

### 14.4. **易于扩展**
- 新增步骤只需添加新的2的幂次方常量
- 不影响现有步骤的完成状态

## 15. 关于30这个值的分析

### 15.1. 二进制分解
```
30的二进制: 11110
分解为: 16 + 8 + 4 + 2 = 30
```

### 15.2. 步骤完成状态
- ✅ **STEP_TAP(16)** - 已完成
- ✅ **STEP_EARN_FOOD(8)** - 已完成
- ✅ **STEP_CLAIM_PRIZE(4)** - 已完成
- ✅ **STEP_COLLECT(2)** - 已完成
- ❌ **STEP_FEED(1)** - 未完成

### 15.3. 引导起点推断
```typescript
// nextPendingStep(30) 的逻辑
STEP_ORDER = [1, 2, 4, 8, 16]  // 按顺序检查

// 检查STEP_FEED(1): (30 & 1) === 1? false → 返回STEP_FEED
```

**结论：后台返回30确实会落到第一步(STEP_FEED)，这是完全符合预期的！**

## 16. 这种设计的业务意义

### 16.1. **支持用户跳过步骤**
- 用户可能因为各种原因跳过某些引导
- 系统能够智能地从第一个未完成的步骤继续

### 16.2. **乱序完成的容错性**
- 即使步骤完成顺序与设计不一致，系统也能正确处理
- 确保用户不会错过任何必要的引导

### 16.3. **调试和测试友好**
- 可以通过设置不同的掩码值来测试各种场景
- 比如设置30来测试"只剩喂食未完成"的情况

## 17. 实际应用场景

### 17.1. 场景1：用户跳过喂食直接收蛋
```
初始状态: 0 (未开始)
用户完成收蛋: 0 | 2 = 2
下次引导: nextPendingStep(2) → 返回STEP_FEED(1)
```

### 17.2. 场景2：用户完成除第一步外的所有步骤
```
状态: 30 (16+8+4+2)
下次引导: nextPendingStep(30) → 返回STEP_FEED(1)
```

**掩码机制的核心价值在于它的灵活性和鲁棒性**，能够适应各种用户行为模式，确保引导系统既不会重复展示已完成的步骤，也不会遗漏未完成的步骤。
